## Incident Pattern: Payment Service Memory Leak After Configuration Change

**Trigger:** "Memory leak in payment service" alert on payment-svc.

**Root Cause:** Recent configuration change introduced a memory leak in payment-svc deployment v3.2.0.

**Immediate Fix (< 5 min):**
1. Identify recent configuration changes associated with payment-svc deployment v3.2.0 (e.g., using deployment history tools or configuration management system).
2. If a clearly problematic configuration change is identified, revert to the previous known good configuration.
3. Monitor memory usage of payment-svc after reverting the configuration.

**Permanent Fix:**
1. Analyze the reverted configuration change to pinpoint the exact cause of the memory leak.
2. Implement a fix for the memory leak in the configuration or code.
3. Deploy the corrected configuration or code to a staging environment for thorough testing.
4. Deploy the corrected configuration or code to production after successful testing in staging.

**Prevention:**
- Implement a robust configuration change management process with peer review and automated testing.
- Improve memory usage monitoring and alerting to detect leaks earlier.
- Implement automated rollback procedures for configuration changes.

**Affected Services:**
- payment-svc

**Escalation:** If not resolved in 30 minutes, page the Payments team.
